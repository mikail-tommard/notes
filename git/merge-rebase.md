# git merge && git rebase

# Описание

Тут мы уже рассмотрим как делать слияние веток, в каких случаях использовать git merge, а в каких случаях использовать git rebase.
По сути своей git merge и git rebase выполняют одну и туже роль, это подтянуть изменения и объединить ветки, но каждый из них это делает по своему.

## git merge

Merge - сохраняет историю как реально все происходило с отвлетвлениям и merge-коммитами. 
Как делается git merge? Предположим у нас есть две ветки, ветка main и ветка feature. Вот мы закончили писать нашу фичу в ветке feature и нам надо подтянуть все в main, делатся это вот так. Сначала мы переходим с ветки feature на ветку main `git checkout main` после уже пишем `git merge feature`, и наши ветки должны слиться. Конечно могут быть конфликты при слиянии, и вам нужно будет эти конфликты исправить.
Хорошей практикой считается, прежде чем отправить ветки на слияние сделать в feature `git pull origin main` - данная команда подтянет изменения из main в наше ветку features если они есть, делается это воизбежание ситуация когда ты вливаешь фичу в устаревшкую локальную версию main.

## git rebase

Rebase - переписывает историю твоей ветки, как будто бы ты начал историю с самой свежей точки.
Как делать git rebase? Возьмем наш предедуший случай с ветками main и feature. Если тебе нужно подтянуть уже новые коммиты которые есть в main, ты пишешь `git fetch` потом `git rebase origin/main` и это должно подтянуть последние коммиты из main.

## В каких случаях что использовать?

Если ты делаешь свою фичу в отдельной локальной ветке features и хочешь туда подтянуть уже новые коммиты из main, спокойно можешь использовать `git rebase`. Но ни в коем случае не делай rebase на те ветки которые запушены и ими пользуются другие, если так сделаешь то у них возможно будут конфликты. Если хочешь сделать слияние в ветку main делай это через `git merge`.
