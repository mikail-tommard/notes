# Отличие между domain сущностями и DTO

## Описание

Вообще суть domain сущностей заключается в том что он должен быть чистым и не зависемым от json и db тегов, то есть независимы от способа доставки.
Где должны лежать DTO структуры. DTO структуры лежат обычно там где они используются, пример:

## DTO

```go
type CreateUserResponce struct {
	ID string `json:"id"`
	Name string `json:"name"`
	Email string `json:"email"`
}
```

В данном случае, данная структура отвечает за то какой ответ мы отправим обратно пользователю, ответы то есть responce у нас отправляется в http.

В чистой архитектуре http/handler мы описываем в директории internal/adapters/http/handler.go но так же можно описывать DTO структуры не прямо в handler.go а в той же директории мы можем создать models.go и описать их там.

Стоит подчеркнуть что DTO храниться не только в http но так же в repository, обмен данными между sql и нашим сервисом так же происходит через DTO структуры.

Зачем их разделять:

1. Чтобы не тянуть детали транспорта/хранения в домен.
2. Изменять API/БД без поломки домена.
3. Контролировать что “можно принимать, а что отдавать”, а что внутренее.

## Domain

Где мы описываем по сути domain сущности, описываются они в директории internal/domain/user.go. Возьмем на примере user.go опишем саму структуру и какая бизнес логика может лежать в domain.

```go
type User struct {
	id string
	name string
	email string
}
```

Описали просто структуру User и описали какие данные он может в себе хранить.

В domain - мы так же описываем бизнес логику, в данном случае тут будет описываться валидация структуры User, конструктор где как раз будет проверяться правильность сущности, изменение имени или почты так же будет храниться в domain.

Пример:

```go
func New(id, name, email string) (*User, error) {
	name = strings.TrimSpace(name)
	email = strings.TrimSpace(email)

	if len(name) < 2 {
		return nil, fmt.Errorf("invalid length name")
	}
	if !strings.Contains(email, "@") {
		return nil, fmt.Errorf("invalid email")
	}

	return &User{
		id: id,
		name: name,
		email: email,
	}, nil
}
```

Данный код это конструктор создания нового User где на месте делается валидация имени и почты, чаще всего именно вот так описываются конструкторы(в разных слоях конструкторы могут или даже выгледят по разному).
